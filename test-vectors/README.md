# Test vectors

## Use `v2/`

[`v2/apertomemory-v2-test-vectors.json`](v2/apertomemory-v2-test-vectors.json)
is the normative artefact for the current format. Fourteen vectors,
regenerated by [`v2/generate.py`](v2/generate.py) and checked by CI on
every push: an artefact that cannot be reproduced from its own generator
is not normative.

**Six of the fourteen are refusal cases.** Reproducing the byte strings
is necessary but not sufficient. An implementation that reproduces every
byte and refuses none of the attacks is not conformant — it is the bug
the vectors exist to catch, and it is what the first release of this
format shipped.

Read [`v2/README.md`](v2/README.md) before implementing.

## Do not use `amem-testvector-001.json` / `-002.json`

Those two are the original `format_version` 1 vectors. They are kept for
historical reference and because vector 009 in the v2 set describes how a
conformant implementation must treat v1 objects.

They encode a format with a known security defect: nothing binds a
signature to the author it claims or to the object envelope around it, so
forged provenance verifies as valid. See
[GHSA-jwrj-j847-ph54](https://github.com/apertomemory/apertomemory/security/advisories/GHSA-jwrj-j847-ph54).

An implementation written against them will be vulnerable in exactly that
way. They are not a starting point.
